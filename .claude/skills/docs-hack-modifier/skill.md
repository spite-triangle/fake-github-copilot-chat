# Docs Hack Modifier

根据 `docs/hack/modules/` 目录下的修改文档，对项目工程中的相关代码进行修改，并生成修改报告。

## 目录结构

```
docs/hack/
├── modules/          # 原始修改文档目录
│   ├── 1_登陆验证.md
│   ├── 2_模型.md
│   └── ...
├── report/           # 修改进度报告目录
│   ├── 1_登陆验证.md
│   └── ...
└── new/              # 反映实际修改的文档目录
    ├── 1_登陆验证.md
    └── ...
```

## 工作流程

### 步骤 1：初始化

1. 确定要处理的修改文档：
   - 如果用户指定了文件名，直接使用（如 `1_登陆验证.md`）
   - 如果用户未指定，列出 `docs/hack/modules/` 下所有可用文档供选择

2. 读取原始修改文档 `docs/hack/modules/{文件名}.md`

3. 创建 `docs/hack/report/` 目录（如不存在）

4. **创建初始报告文件** `docs/hack/report/{文件名}.md`：
   - 报告初始状态为"待修改"
   - 记录每���修改点的初始状态

### 步骤 2：分析修改描述

仔细阅读修改文档，提取以下信息：

1. **修改点列表**：每个修改点包含：
   - 目标文件路径
   - 目标类/函数/接口
   - 修改类型（新增、修改、删除）
   - 具体修改内容（代码片段）

2. **识别代码块类型**：
   - TypeScript 接口修改
   - 函数实现重写
   - 构造函数修改
   - 字段/属性添加

3. **识别所有需要的导入**：
   - 检查代码中使用的 `workspace` 是否需要导入
   - 检查 `RequestType`、`RequestMetadata` 等是否需要导入
   - 检查其他 vscode API 或外部模块是否需要导入

4. **识别配置项命名**：
   - 特别注意配置项名称（如 `retry.resp_max`），**不要自行修改**
   - 识别配置项的完整参数（次数 + 延迟）

5. **更新报告**：在报告中记录每个修改点的"待处理"状态

### 步骤 3：检查目标文件

对于每个修改点：
1. 验证目标文件是否存在
2. 验证目标类/函数/接口是否存在
3. 在报告中记录检查结果

### 步骤 4：执行代码修改

按以下优先级处理修改：

**优先级 A - 完全匹配**：目标文件和内容都存在
- 直接按照文档描述进行修改
- 更新报告状态为"✅ 完成"

**优先级 B - 部分匹配**：目标文件存在但内容不完全匹配
- 尽量按照修改意图进行修改
- 在报告中说明差异，更新状态为"⚠️ 部分完成"

**优先级 C - 缺失内容**：目标文件或内容不存在
- 跳过该修改
- 在报告中标记为"❌ 无法完成"
- 说明原因

### 步骤 5：生成 new 目录文档

**触发条件**：每次修改完成时均需生成

**new 文档的作用**：
- 反映实际实现的修改结果
- 作为下次修改的基础文档（迭代式修改）
- 包含完整的修改内容，便于后续版本参考和执行修改

**核心原则**：
- **new 文档格式必须与 `docs/hack/modules/` 中原文档保持一致**
- 一个小节（`##`）为一个修改点
- 每个修改点的描述必须准确，能指导后续版本的修改

**操作方式**：
1. 读取 `docs/hack/modules/{文件名}.md`（原版文档）
2. 读取 `docs/hack/new/{文件名}.md`（如果存在，作为当前基准）
3. 读取实际修改后的代码文件
4. 创建/更新 `docs/hack/new/{文件名}.md`

**new 文档结构规范**：

每个修改点（小节）必须包含以下信息：

```markdown
## 修改点 N
- **文件**: 目标文件路径（相对于项目根目录）
- **位置**: 具体位置，如类名、方法名、函数名

### 修改前
（修改前的原始代码，如有）

### 修改后
（修改后的实际代码）

### 说明
（修改的目的、差异说明或其他备注）
```

**文档整体结构**：
```markdown
# {标题}

## 说明
描述本次修改的背景和目的。

---

## 修改点 1
- **文件**: xxx
- **位置**: xxx

### 修改前
```ts
// 原始代码
```

### 修改后
```ts
// 修改后代码
```

### 说明
（可选）

---

## 修改点 2
...

---

## 修改完成时间
YYYY/MM/DD
```

**注意事项**：
- new 文档专注于修改内容的描述，**不包含修改类型、状态等元信息**（这些属于 report 文档）
- new 文档应完整保留原版文档的所有修改点描述
- 代码块必须使用正确的语言标识（如 `ts`、`js`、`json` 等）
- 如果原版文档中的某个修改点被跳过或无法完成，仍需在 new 文档中保留该小节
- new 文档作为下一次修改的起点，形成迭代式文档管理

## 注意事项

1. **保持修改完整性**：尽量与修改文档保持一致，不要随意删减修改内容

2. **尊重现有代码**：优先在现有代码基础上修改，而非完全替换

3. **处理缺失内容**：
   - 被修改内容在项目中已不存在 → 跳过修改，在报告中标记
   - 无法按照描述修改 → 标记原因

4. **代码格式**：保持与项目一致的缩进和格式风格

5. **报告同步**：每次修改后都应更新报告文件，保持状态同步

6. **配置项命名**：
   - **绝对不要自行修改配置项名称**，必须与原文档完全一致
   - 配置项要包含完整参数（次数 + 延迟）

7. **导入语句**：
   - 检查所有 `workspace.getConfiguration` 是否需要导入 `workspace`
   - 检查 `RequestType`、`RequestMetadata` 等是否需要导入
   - 按需添加，不要遗漏

## 示例

**输入**: 用户说"根据 docs/hack/modules/1_登陆验证.md 进行修改"

**执行流程**:
```
1. 读取 docs/hack/modules/1_登陆验证.md
2. 创建 docs/hack/report/1_登陆验证.md（初始状态）
3. 分析 3 个修改点（加快激活、session、token）
4. 检查 3 个目标文件是否存在
5. 执行代码修改
6. 如有差异，更新 docs/hack/new/1_登陆验证.md
```

**输出**: 确认修改完成，提供报告文件路径
